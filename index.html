HIFI FILTER : filtration et séparation

HIFI FILTER est une entreprise spécialisée dans les solutions de filtration et de séparation pour les équipements mobiles et les procédés industriels. Son activité consiste à développer et distribuer une large gamme de filtres, accessoires et kits de maintenance pour accompagner les professionnels dans l'entretien de leurs machines, véhicules et installations.

Les produits couvrent plusieurs domaines : filtration moteur, hydraulique, air comprimé, dépoussiérage, machines-outils, liquides, ventilation ou encore produits complémentaires. L'entreprise s'adresse donc à des secteurs variés comme l'agriculture, les travaux publics, le transport, l'industrie ou les process de production.

Dans ce contexte, les outils informatiques internes ont un rôle important : ils permettent de gérer les données métier, les applications utilisées par les équipes, les processus de support, les contrôles et la fiabilité des environnements applicatifs.

Mon stage se déroule dans l'équipe IT Helios, une équipe orientée développement et maintenance d'applications internes. Elle travaille principalement autour de l'écosystème Microsoft : Dynamics 365, Dataverse, Power Apps, Power Automate, ainsi que des outils internes développés en Python.

L'équipe fonctionne avec une organisation agile de type Scrum : le travail est découpé en tickets, suivi sur Azure DevOps, puis rythmé par des sprints, des points de planification, des rétrospectives et des revues. Les développements passent aussi par des branches, des pull requests et des retours de revue avant d'être validés.

Contexte du stage

Un sujet centré sur la fiabilisation des outils internes

Le sujet principal de mon stage est de contribuer à la fiabilisation d'outils utilisés par l'IT. Une partie du travail concerne des contrôles d'intégrité sur Power Platform : comparaison d'environnements, détection d'écarts, vérification de rôles de sécurité, contrôle des layers non managées ou génération de rapports exploitables par l'équipe.

L'autre partie concerne DataCuddle, un outil Python interne utilisé pour manipuler des workflows et automatiser des tests. J'ai travaillé sur l'évolution de son moteur, l'ajout de conditions, de fonctions, de boucles, de logs et sur une réflexion autour de tests Python plus simples à maintenir que des workflows JSON.

L'enjeu commun de ces travaux est le même : réduire les erreurs difficiles à détecter, améliorer la visibilité sur les écarts entre environnements et rendre les outils plus fiables pour les développeurs et les équipes métier.

4 traces techniques
Trace 1

Ajouter un moteur de conditions dans DataCuddle

Résoudre une expression Exécuter une branche Propager les erreurs Corriger après revue

Descriptif général de la trace

Cette trace montre l'ajout d'une action condition dans DataCuddle. Avant cette évolution, un workflow exécutait surtout une suite d'étapes de manière linéaire. Avec cette action, il peut maintenant évaluer une expression puis choisir une branche true ou false.

La difficulté était d'intégrer cette logique dans un moteur existant sans casser les autres actions. Il fallait aussi conserver un résultat lisible : si une sous-étape échoue, la condition doit remonter un statut exploitable et un détail d'erreur compréhensible.

Analyse des savoir-faire

Pour résoudre une expression, j'utilise les variables partagées du workflow avec resolve_condition_expression, puis je transforme le résultat en booléen avec evaluate_resolved_condition_expr.

Pour exécuter une branche, le moteur relance run_workflow_steps sur la liste de steps de la branche choisie. Cela permet de conserver la même logique d'exécution que pour un workflow classique.

Pour propager les erreurs, j'ai dû faire remonter l'échec d'une sous-step vers la condition. Ainsi, le workflow ne donne pas un résultat trompeur si une étape enfant échoue.

Pour corriger après revue, les remarques de Nicolas m'ont amené à éviter des traitements inutiles et à améliorer le détail retourné en cas d'échec.

Trace 2

Fiabiliser les fonctions d'expression

Implémenter des fonctions Gérer les types Traiter les cas limites Compléter les tests

Descriptif général de la trace

Cette trace montre l'évolution de l'interpréteur d'expressions DataCuddle. Le besoin était d'ajouter des fonctions utilisables dans les expressions, comme select(), map(), filterArray(), add(), sum(), greater(), less() ou equals().

Ce travail demandait plus que l'ajout de fonctions : il fallait définir un contrat clair pour chaque type de valeur. Une comparaison entre nombres, dates ou chaînes ne doit pas produire un résultat incohérent ou silencieux.

Analyse des savoir-faire

Pour implémenter des fonctions, j'ai ajouté des comportements réutilisables dans le moteur d'expressions. Ces fonctions devaient rester simples à appeler, mais prévisibles dans leur résultat.

Pour gérer les types, la revue a été importante. Les nombres, dates ISO, booléens, chaînes et valeurs None ne peuvent pas tous être comparés de la même manière.

Pour traiter les cas limites, j'ai dû préciser le comportement de string(null), des comparaisons avec None et des champs manquants dans les fonctions comme select().

Pour compléter les tests, les retours de revue ont amené à ajouter des cas comme equals("a", "b"), la différence de casse ou la sélection d'une clé inexistante.

Trace 3

Remplacer des workflows JSON par des tests Python

Concevoir une API de test Utiliser pytest Sécuriser setup / teardown Gérer les environnements 
# Extrait de conftest.py
	@pytest.fixture(scope="session")
	def dc(request):
	    return datacuddle.DCManager(
	        env=request.config.getoption("--env"),
	        mode="line",
	        nb_workers=2,
	        skip_plugin=True,
	        output_dir=request.config.getoption("--output-dir"),
	        impersonate=request.config.getoption("--impersonate"),
	        test_dir="unitTest/DataCuddle - CRUD",
	    )

Descriptif général de la trace

Cette trace montre le changement d'approche décidé autour de DataCuddle. Au lieu de continuer à décrire certains tests sous forme de workflows JSON, l'équipe a commencé à utiliser des tests Python avec pytest et une façade DCManager.

L'objectif était de rendre les tests plus naturels à écrire et plus faciles à maintenir. Les actions DataCuddle deviennent des appels Python explicites comme dc.insert(), dc.update(), dc.verify() ou dc.delete().

Analyse des savoir-faire

Pour concevoir une API de test, il faut que les appels restent lisibles : dc.insert(), dc.update(), dc.verify(), dc.delete(). Le test raconte alors clairement le scénario exécuté.

Pour utiliser pytest, je m'appuie sur une fixture qui prépare DCManager avec les options communes. Cela évite de répéter la configuration dans chaque test.

Pour sécuriser setup / teardown, le bloc finally est indispensable : même si une vérification échoue, les données de test sont nettoyées pour ne pas polluer l'environnement Dataverse.

Pour gérer les environnements, la fixture récupère des options comme --env, --output-dir ou --impersonate. Le même test peut donc être adapté à plusieurs contextes d'exécution.

Trace 4

Automatiser le contrôle des unmanaged layers

Interroger Dataverse Filtrer et regrouper Produire un rapport Teams Rendre l'alerte exploitable

Descriptif général de la trace

Cette trace montre un flow Power Automate utilisé pour détecter des layers non managées dans Dataverse. Le flow est lancé manuellement, cible un environnement, liste les composants de solutions HIFI et identifie ceux qui présentent une anomalie.

Le résultat est publié dans Teams sous forme d'alerte. Cela rend le contrôle directement exploitable par l'équipe : les éléments détectés peuvent être corrigés, puis suivis dans le fil de discussion.

Analyse des savoir-faire

Pour interroger Dataverse, j'utilise des actions HTTP ou List Rows avec les bons champs : solution, composant, type de composant et parent éventuel.

Pour filtrer et regrouper, le flow utilise des étapes Filter array, Select et Apply to each. Le but est de transformer des lignes Dataverse en liste lisible par solution.

Pour produire un rapport Teams, je construis un message directement actionnable. La trace n°4B montre une alerte publiée dans Teams avec les composants concernés.

Pour rendre l'alerte exploitable, le message doit être assez précis pour permettre à l'équipe de savoir quoi corriger et de suivre la résolution après déploiement.
